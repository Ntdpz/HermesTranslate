### 1. System Infrastructure (โครงสร้างพื้นฐานและเทคโนโลยี)

* **API Gateway:** พัฒนาตัวรับ Request ด้วยภาษา **Python** โดยใช้ Asynchronous Framework (เช่น **FastAPI**) เพื่อให้สามารถรับโหลดและจัดการ Concurrency (การทำงานหลายอย่างพร้อมกันในเวลาเดียวกัน) ได้อย่างมีประสิทธิภาพ
* **Deployment:** ระบบตัวแม่ `Hermes Main Agent` และ Worker ทั้งหมดจะต้องถูกแพคและรันอยู่บน **Docker** (แพลตฟอร์มจำลองสภาพแวดล้อมการทำงานของซอฟต์แวร์ให้อยู่ในคอนเทนเนอร์) เพื่อให้สเกลระบบได้ง่าย
* **Database:** ใช้ **PostgreSQL** ในการจัดเก็บสถานะของงานและกฎการแปลต่างๆ

### 2. Core Workflow (การไหลของข้อมูลแบบ Asynchronous)

ระบบนี้ห้ามทำแบบรอผลลัพธ์หน้าจอเด็ดขาด ต้องเป็น **Asynchronous API** (รูปแบบการทำงานที่เซิร์ฟเวอร์รับคำสั่งแล้วคืนสถานะ 'กำลังดำเนินการ' ทันที โดยไม่บล็อคการทำงานอื่น)

* **Step 1:** หน้าบ้านส่งข้อมูลเข้า API
* **Step 2:** API รับคำสั่ง คืนค่า `Task_ID` ให้หน้าบ้านทันที แล้วโยนข้อมูลลง **Message Queue** (ระบบคิวข้อความที่ช่วยพักและจัดลำดับงานเบื้องหลัง เช่น RabbitMQ)
* **Step 3:** หน้าบ้านเอา `Task_ID` ไปทำ **Polling** (เทคนิคที่ไคลเอนต์ตั้งเวลาเพื่อยิงคำขอไปถามเซิร์ฟเวอร์เป็นระยะๆ ว่างานเสร็จหรือยัง) หรือรอรับ **Webhook** (การที่เซิร์ฟเวอร์ปลายทางยิงข้อมูลกลับมาอัตโนมัติเมื่องานเสร็จ)

### 3. Knowledge Base & Rule Management (ระบบสมองและกฎการสอน AI)

* **UI & DB:** มีหน้า Backoffice ให้ Admin เข้าไปเพิ่ม/ลด/แก้ไข "กฎเฉพาะทาง" และเก็บลง PostgreSQL
* **Rule Filtering:** ใช้ **Aho-Corasick Algorithm** (อัลกอริทึมที่ใช้ค้นหาคำหลายๆ คำในข้อความยาวๆ ได้พร้อมกันในรวดเดียวแบบเร็วจัดๆ) เพื่อสแกนหา Exact Match (คำที่ตรงกันเป๊ะๆ) ในประโยคที่ User พิมพ์มา จะได้ดึงเฉพาะกฎที่เกี่ยวข้องจริงๆ ไปใส่ Prompt ลดปัญหาผลาญ Token
* **Conflict Resolution** (การจัดการข้อขัดแย้ง): หาก Admin เผลอตั้งกฎที่ขัดแย้งกันเอง ระบบจะยึดกฎที่ถูกอัปเดตล่าสุด (Timestamp ล่าสุด) เป็นหลักเสมอ

### 4. Multi-Agent System Engine (กลไกการทำงานด้วย MD Template ควบคุม 3 Agents)

ระบบใช้การควบคุม Workflow ด้วย **MD Template** ในการส่งต่อข้อมูลและคำสั่ง แทนการใช้เฟรมเวิร์กที่ซับซ้อน โดยมี Agent เพียง 3 ตัวทำงานประสานกัน และส่วน Frontend คอยติดตามสถานะแบบเบื้องหลัง (Background Monitoring)

* **Step 4.1 - `Main Agent` (Orchestrator):** ดึงงานจากคิว -> สแกนข้อความด้วย Aho-Corasick -> ทำ **Dynamic Prompt Injection** ลงใน MD Template -> ส่งมอบงานให้ Translate Agent
* **Step 4.2 - `Translate Agent` (Creator):** รับข้อมูล MD Template จาก Main Agent แล้วทำการแปลข้อความหรือสร้างเนื้อหา จากนั้นส่งผลลัพธ์ต่อไปยัง Validate Agent
* **Step 4.3 - `Validate Agent` (Validator):** รับผลลัพธ์มาตรวจทานความถูกต้อง โดยใช้กฎชุดเดียวกับที่ Main Agent ดึงมา เพื่อเช็คว่า Translate Agent ทำตามคำสั่งใน MD Template หรือไม่

### 5. Fail-safes & Fallback (ระบบป้องกันความพินาศ)

* **Max Retries:** กรณีที่ `Validate Agent` ตรวจเจอจุดผิด ระบบจะตีกลับ (Retry) งานไปหา `Translate Agent` เพื่อแก้ใหม่ ด้วยลอจิกโค้ดภายในระบบ (โดยไม่พึ่งพา State Machine ภายนอก) โดยจำกัดไว้ที่ **สูงสุด 3 รอบเท่านั้น** เพื่อป้องกัน **Infinite Loop** (ข้อผิดพลาดของโปรแกรมที่ทำให้เกิดการทำงานวนซ้ำไปเรื่อยๆ) ถ้ารอบที่ 3 ยังผิด ให้หยุดการทำงานและปรับสถานะเป็น "Failed/Manual Review" ทันที
* **Idempotency** (คุณสมบัติการทำงานที่สั่งซ้ำกี่ครั้งก็ได้ผลลัพธ์เท่าเดิม): ทุก Request ต้องมีการเช็ค `Task_ID` ซ้ำเสมอ ป้องกันการรัน AI เบิ้ลในกรณีที่เน็ตเวิร์กกระตุก
