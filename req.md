เอาล่ะ Junior เตรียมสมุดจดให้ดี พี่จะรวบยอด Requirement ทั้งหมดตั้งแต่โครงสร้างพื้นฐานไปจนถึงระบบสอน AI ที่เพิ่งงอกมาใหม่ โครงสร้างนี้ถือว่าทะเยอทะยานและตอบโจทย์สเกลระดับโปรดักชันได้ดีมาก พี่ขอชมว่าวิสัยทัศน์การออกแบบโดยรวมถือว่ายอดเยี่ยมและครอบคลุม แต่ถ้าโค้ดดิ้งไม่รัดกุม ระบบคุณจะพังพินาศได้ง่ายๆ

นี่คือ **System Requirement Specification** ฉบับไฟนอลที่คุณต้องเอาไปแตก Task ให้ทีม Dev ทำตามนี้เป๊ะๆ ห้ามหลุดแม้แต่ข้อเดียว:

---

### 1. System Infrastructure (โครงสร้างพื้นฐาน)

* **Deployment:** ระบบตัวแม่ `Hermes Main Agent` และ Worker ทั้งหมดจะต้องถูกแพคและรันอยู่บน **Docker** (แพลตฟอร์มจำลองสภาพแวดล้อมการทำงานของซอฟต์แวร์ให้อยู่ในคอนเทนเนอร์ เพื่อให้สามารถนำไปรันที่ไหนก็ได้โดยระบบไม่พัง) เพื่อให้ง่ายต่อการเพิ่มลดขนาดระบบเวลามีคนใช้งานเยอะ

### 2. Core Workflow (การไหลของข้อมูล)

ระบบนี้ห้ามทำแบบรอผลลัพธ์หน้าจอเด็ดขาด ต้องเป็น **Asynchronous API** (รูปแบบการทำงานที่เซิร์ฟเวอร์รับคำสั่งแล้วคืนสถานะ 'กำลังดำเนินการ' ทันที โดยไม่บล็อคการทำงานอื่น)

* **Step 1:** หน้าบ้านส่งข้อมูลเข้า API
* **Step 2:** API รับคำสั่ง คืนค่า `Task_ID` ให้หน้าบ้านทันที แล้วโยนข้อมูลลง **Message Queue** (ระบบคิวข้อความที่ช่วยพักและจัดลำดับงานเบื้องหลัง เช่น RabbitMQ หรือ Kafka)
* **Step 3:** หน้าบ้านเอา `Task_ID` ไปทำ **Polling** (เทคนิคที่ไคลเอนต์ตั้งเวลาเพื่อยิงคำขอไปถามเซิร์ฟเวอร์เป็นระยะๆ ว่างานเสร็จหรือยัง) หรือรอรับ **Webhook** (การที่เซิร์ฟเวอร์ปลายทางยิงข้อมูลกลับมาหาต้นทางโดยอัตโนมัติทันทีที่งานเสร็จ)

### 3. Knowledge Base & Rule Management (ระบบสมองและกฎการสอน AI)

นี่คือฟีเจอร์ใหม่ที่คุณเพิ่งเพิ่มเข้ามา ต้องทำสิ่งนี้ก่อน AI จะเริ่มทำงาน:

* **UI & DB:** ต้องมีหน้า Backoffice ให้ Admin เข้าไปเพิ่ม/ลด/แก้ไข "กฎเฉพาะทาง" ได้ และเก็บลง Database (เช่น คำนี้ให้ทับศัพท์, คำนี้ห้ามแปลตรงตัว)
* **Conflict Resolution** (การจัดการข้อขัดแย้ง): หาก Admin เผลอตั้งกฎที่ขัดแย้งกันเอง ระบบต้องมีลอจิกในการเลือกกฎ โดยยึดจาก Priority หรือกดยึดกฎที่ถูกอัปเดตล่าสุด (Timestamp) เป็นหลักเสมอ

### 4. Multi-Agent System Engine & LangGraph (กลไกการทำงานของ AI)

ให้แบ่งหน้าที่ **Separation of Concerns** (หลักการออกแบบซอฟต์แวร์ที่แบ่งระบบออกเป็นส่วนๆ ตามหน้าที่รับผิดชอบที่ชัดเจน ไม่ก้าวก่ายกัน) โดยใช้ **LangGraph** มาเป็นโครงสร้างหลัก (State Machine) ในการควบคุม Flow การทำงาน (Node และ Edge) และจัดการ State ทั้งหมด ดังนี้:

* **State Management & Checkpointing:** ทุกขั้นตอนต้องมีการเก็บสถานะการทำงานผ่านระบบ State ของ LangGraph และบันทึก Checkpoint อย่างสม่ำเสมอ (เผื่อกรณีระบบล่มหรือคอนเทนเนอร์ดับ สามารถกลับมารันต่อจาก Node ล่าสุดได้ทันทีโดยไม่ต้องเริ่มต้นกระบวนการใหม่)
* **Node Timeout:** ทุก Node ต้องมีการกำหนด Timeout ให้ชัดเจน เพื่อป้องกันระบบค้างในกรณีที่ API ของ LLM ไม่ตอบสนอง

* **Step 4.1 - The Orchestrator (`Hermes Main Agent`):**
รับงานจาก Queue -> นำข้อมูลเข้าสู่ LangGraph State -> สแกนข้อความ User ว่าตรงกับกฎในข้อ 3 หรือไม่ -> ทำ **Dynamic Prompt Injection** (การดึงข้อมูลจากฐานข้อมูลมาต่อท้ายหรือแทรกในคำสั่ง AI แบบอัตโนมัติ) เพื่ออัปเดตคำสั่งที่สมบูรณ์ลงใน State ก่อนส่งต่อ
* **Step 4.2 - The Creator (`Sun Agent`):**
รับคำสั่งและกฎบังคับจาก Hermes แล้วปั่นเนื้อหา (ชื่อ TH/EN, รายละเอียด TH/EN) ออกมาในรูปแบบ **Structured JSON** (รูปแบบข้อมูลที่จัดเรียงเป็น Key-Value คู่กัน เพื่อให้โปรแกรมนำไปใช้ต่อได้ง่าย) เท่านั้น ห้ามตอบเป็นข้อความธรรมดา
* **Step 4.3 - The Validator (`CheckWordAgent`):**
เอา JSON ที่ได้ มาตรวจทานความถูกต้อง **โดยใช้กฎ (Rules) ชุดเดียวกับที่ Hermes ดึงมา** เพื่อเช็คว่า Sun Agent ทำตามคำสั่งเป๊ะหรือไม่

### 5. Fail-safes & Fallback (ระบบป้องกันความพินาศ)

* **Max Retries via LangGraph:** (จำนวนรอบสูงสุดที่อนุญาตให้ระบบทำงานซ้ำ) กรณีที่ `CheckWordAgent` ตรวจเจอจุดผิด ให้ใช้ Conditional Edge ใน LangGraph ตีกลับไปหา `Sun Agent` (วงจรลูปย้อนกลับ) เพื่อแก้ใหม่ แต่ต้องจำกัดไว้ที่ **สูงสุด 3 รอบเท่านั้น** (นับจำนวน Retries เก็บไว้ใน State) เพื่อป้องกัน **Infinite Loop** (ข้อผิดพลาดของโปรแกรมที่ทำให้เกิดการทำงานวนซ้ำไปเรื่อยๆ ไม่มีจุดสิ้นสุด) ถ้ารอบที่ 3 ยังผิดอยู่ ให้โยนสถานะ "Failed/Manual Review" ลง Database แล้วชี้ไปยัง End Node เพื่อหยุดการทำงานทันที
